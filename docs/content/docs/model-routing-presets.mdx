# Model Routing Presets

Cutctx supports an opt-in named model-routing preset through `CUTCTX_MODEL_ROUTING_PRESET`.

## Canonical preset

- `codex-gpt54mini-high`

## Compatibility aliases

- `codex-opencode-slim`
- `oh-my-opencode-slim`

## Evidence-gated Claude three-tier preset

`claude-three-tier-eval` is an explicit opt-in preset for an
Opus→Sonnet→Haiku graph:

- low-complexity Opus requests may route to Haiku
- medium/contextual Opus requests may route to Sonnet
- high-risk requests remain on Opus
- low-complexity Sonnet requests may route to Haiku

This preset sets `require_calibrated_scorer=true`. Without a valid promoted
scorer artifact, every decision abstains with `calibrated_scorer_required`.
The canonical `codex-gpt54mini-high` preset retains its existing conservative
Claude behavior and does not enable this second downgrade.

## Behavior

The preset is intentionally conservative:

- High-complexity GPT-5-tier and Claude-tier work stays on the originally requested model.
- Low-complexity GPT tasks route to `gpt-5.4-mini`.
- Contextual and moderate GPT-5.5/5.6 tasks route to `gpt-5.6-luna` when the route declares it as the medium tier.
- Low-complexity Claude tasks route to the next cheaper Claude tier.
- Routed low-complexity requests also receive `reasoning: { effort: "high" }` on OpenAI request paths.
- The canonical preset explicitly allows `gpt-5.4-mini` and `gpt-5.6-luna` on Codex/ChatGPT subscription transports; arbitrary custom targets remain blocked unless explicitly allowlisted.

## Aggressive preset coverage

The dashboard's **Aggressive** mode uses the `economy` preset. It has
explicit exact-match coverage for the current project model labels, including
`gpt-5.6-terra`, `gpt-5.6-sol`, `gpt-5.6-luna`, `gpt-5.5`, `gpt-5.4`,
`gpt-5.3`, `gpt-5.2`, `gpt-5.1`, `gpt-5`, `claude-sonnet-5`, and their
documented versioned Claude aliases. GPT-5 models route to `gpt-5.4-mini`;
Sonnet 5 routes to `claude-haiku-5-20260101` when the request is eligible.

The router intentionally does not use family-name wildcards. An unlisted
provider model is retained and recorded as `no_route_for_model` until a
provider-validated exact route is added.

## Seeing why a request stayed put

Recent requests now include a **Routing** value alongside the effective model.
It shows the source-to-target pair for an applied route, or a reason such as
`no_route_for_model`, `workload_not_downgradeable`,
`confidence_below_threshold`, or `downgrade_blocked_unproven_transport`.
Large agent turns with recent tool context, code, or other high-risk signals
are intentionally retained on the requested model even in Aggressive mode.

## Complexity heuristic

The router only applies the low-complexity branch when a request is clearly simple, such as:

- short docstring or type-hint edits
- lint fixes and formatting-only requests
- small renames
- obvious one-file or one-function cleanups

If the task looks ambiguous, broad, or risky, Cutctx keeps the original requested model.

Reference-dependent follow-ups and moderate transformations use the Luna tier. Architecture,
security, debugging, implementation, migration, release, deployment, production, patch, and
large multi-module requests stay on the requested strong model.

Short prompts are not assumed to be easy. Database/schema work, concurrency,
credentials and permissions, destructive operations, performance investigations,
and compound multi-action execution stay on the strong tier even when the request
is only a few words. Conversely, an old tool result does not permanently pin a
long-running conversation to the strong model: only the recent context window is
considered for the tool-context safety gate.

The recent-context gate recognizes provider-native tool activity across Anthropic
`tool_use`/`tool_result` blocks, OpenAI Chat Completions `tool_calls`, and OpenAI
Responses call/output items such as `function_call` and `local_shell_call_output`.
Calls and results in that bounded window keep the requested strong model even when
the latest user text would otherwise qualify for Mini.

The deterministic benchmark at `benchmarks/model_routing_quality.py` gates Mini, Luna, and
strong-model assignments with zero tolerated unsafe Mini downgrades.

The versioned v2 corpus contains generic cases plus dedicated Codex, Claude,
and OpenCode cases covering repository lookups, small edits, contextual
follow-ups, tool results, code blocks, multimodal content, security,
production, migration, debugging, and release work. CI reports safety and tier
accuracy globally, per client, and per category. Every client and category has
zero tolerated unsafe Mini downgrades.

## Confidence and abstention

### Capability-safe downgrades

Before an optimization route changes a model, ingress adapters derive the request's
provider-neutral capabilities (such as tool calling, structured output, vision,
audio, and streaming). A feature-bearing request stays on its requested model
unless the selected target route explicitly declares every required capability.
The routing trace records `target_missing_capabilities` and the missing names.

Provider validation errors are terminal `invalid_request` failures: they are never
retried against another deployment and do not place a deployment into cooldown.

Routing USD savings include both input and output token price deltas when complete
usage and pricing are available. If either output value is unavailable, savings
remain an input-only estimate.

Every applied preset decision carries a scorer name, confidence value, and a
small list of non-sensitive decision signals in routing metadata (for example,
`explicit_low_complexity`, `context_dependent`, or `recent_tool_context`). The
built-in scorer is deterministic and preserves the existing
complexity rules. Operators using custom JSON routes may set
`minimum_confidence` in `CUTCTX_MODEL_ROUTING`; a decision below that threshold
does not downgrade and is recorded as `confidence_below_threshold`.

The scorer interface is intentionally pluggable so deployments can add a
calibrated or trained classifier without bypassing the existing safety gates.

## Offline shadow calibration

Model-routing shadow comparisons are stored separately from normal request and
savings telemetry. Each JSONL row contains only a prompt fingerprint, model
pair, scorer/confidence, normalized quality score, and measured costs; prompt
and response text are not persisted. Override the default
`.cutctx/model_routing_evals.jsonl` location with
`CUTCTX_MODEL_ROUTING_EVAL_PATH`.

Shadow baseline execution is disabled by default. Compatibility handlers may
opt in through `CUTCTX_MODEL_ROUTING_SHADOW_MODE=true` and set a stable sampled
fraction with `CUTCTX_MODEL_ROUTING_SHADOW_SAMPLE_RATE`. Transport-specific
replay supplies the baseline response and provider-reported cost to the shared
failure-isolated evaluator; an evaluation failure never changes the primary
response.

Generate the observed quality/cost frontier and a recommended confidence
threshold with:

```bash
python3 -m benchmarks.model_routing_calibrate \
  .cutctx/model_routing_evals.jsonl \
  --minimum-mean-quality 0.90 \
  --maximum-unsafe-rate 0.01 \
  --minimum-segment-samples 20
```

The command returns no recommendation when the evidence cannot satisfy the
configured quality limits. Shadow execution remains opt-in and never replaces
or changes the primary response. Once the candidate response completes, the
non-streaming hooks schedule sampled baseline evaluation as retained best-effort
background work. The visible response does not wait for the baseline provider
call or local judge.

The report also recommends thresholds independently by client, task type,
source/target model pair, and hashed workspace/repository identity. Client and
task labels are normalized low-cardinality strings; workspace and repository
values are stored only as SHA-256 hashes. A segment below the configured sample
minimum inherits the global recommendation. A sufficiently populated segment
that fails its quality limits is marked `quality_blocked` rather than silently
receiving a weaker segment policy.

Authenticated operators can inspect the same privacy-safe evidence through
`GET /v1/orchestration/routing/evidence`. Its versioned response reports one of
four readiness states:

- `no_evidence`: no shadow comparisons have been recorded
- `collecting`: evidence exists but has not reached the minimum sample requirement
- `quality_blocked`: enough evidence exists, but no threshold satisfies the quality limits
- `ready`: a confidence threshold satisfies the configured quality and unsafe-rate limits

The endpoint defaults to 20 samples, mean quality of at least `0.90`, unsafe
rate of at most `0.01`, and an unsafe quality floor of `0.80`. Query parameters
provide read-only simulation of different constraints; they do not promote a
scorer or mutate live routing. Responses never contain prompts, generated
responses, raw workspace or repository identities, credentials, or the local
evidence path.

### Transport coverage

- Direct Anthropic `/v1/messages`, OpenAI `/v1/chat/completions`, and OpenAI
  `/v1/responses` non-streaming requests support background sampled baseline replay.
- Streaming and WebSocket requests are not replayed. A second live stream can
  consume tools or produce side effects, and switching or joining streams after
  output starts would violate the proxy's transport contract.
- ChatGPT subscription and Codex Responses Lite requests produce shadow
  evidence only if the normal routing layer first proves and applies a
  transport-safe target. A blocked or would-be assignment is never replayed.
- Role/account/provider transport checks run before shadow eligibility; shadow
  mode cannot relax strict orchestration policy.

## Calibrated scorer artifacts

Shadow evidence includes a bounded structural feature vector—request length,
conversation depth, and boolean code/tool/reference/high-risk signals. These
features contain no prompt text and are used to fit a small logistic confidence
model:

```bash
python3 -m benchmarks.model_routing_train \
  .cutctx/model_routing_evals.jsonl \
  --output .cutctx/model_routing_scorer.json \
  --minimum-samples 20 \
  --minimum-segment-samples 20 \
  --minimum-mean-quality 0.90 \
  --maximum-unsafe-rate 0.01
```

Training uses a deterministic holdout. An artifact is rejected unless its
holdout policy satisfies the configured quality limits and matches or improves
the heuristic policy's unsafe rate and measured savings. Enable a promoted
artifact with:

```bash
export CUTCTX_MODEL_ROUTING_SCORER_ARTIFACT=.cutctx/model_routing_scorer.json
```

At runtime, a promoted model-pair threshold takes precedence over a promoted
client threshold, which takes precedence over the global artifact threshold.
Segments without sufficient evidence inherit the global threshold. The
operator-configured `minimum_confidence` remains an absolute lower bound and
cannot be weakened by an artifact.

The learned score controls confidence and abstention only. The deterministic
heuristic still assigns the tier and always retains high-risk work on the
requested model. Missing, malformed, or incompatible artifacts fall back to
the heuristic scorer.

## Unified decision trace

Both optimization presets and explicit role orchestration emit
`model_routing_trace` metadata using schema version 1. The shape is stable
whether a model was changed, retained, rejected by policy, or blocked by the
transport. It includes:

- routing mechanism, requested/assigned/effective models, reason, and applied state
- role, binding, mode, policy, provider, and account when orchestration is used
- scorer and confidence when optimization routing is used
- ordered candidates, rejected candidates and reasons, required capabilities,
  attempted deployments, and fallback history
- provider/account/wire-mode proof and the compatible transport's safe-target state

Transport mismatch errors carry the same trace on
`RoutingUnavailableError.decision_trace`, so fail-closed decisions remain
auditable even though the request is rejected. Shadow evaluation rows use the
same request ID, allowing quality/cost evidence to be joined to the original
decision without storing prompt or response text.

## Harness compatibility

The Orchestrator's **Harnesses** tab reports the supported gateway contracts
for Codex, Claude Code, and OpenCode. A supported harness does not authorize
cross-provider routing by itself: the **Models** tab must show an enabled,
eligible deployment and the proxy must have verified adapter compatibility for
the request protocol before a route can execute.

## Workload contracts and legacy routing

Routing Studio presents workload contracts as the operator-facing source of
intent. A contract names the workload, hard eligibility requirements, an
optimization objective, a reliability budget, and the evidence required for
rollout. The compiler turns that contract into the existing deterministic role
and binding configuration, so capability, provider, account, residency, data
classification, budget, and transport checks still run before optimization.

Existing roles and bindings remain compatible. When no contract versions have
been stored, the service exposes read-only `active` legacy contracts that map
each role to its current baseline and fallback chain without writing migration
state. Saving the first draft materializes the new versioned control plane.

The route simulator evaluates the visible draft rather than the active server
configuration. Preview and live decisions share schema-v2 receipts, including
contract identity, compiled policy hash, candidate rejection reasons, selected
model and deployment, evidence source, and the applicable reliability budget.
Provider availability and fallback success are reliability evidence, not answer
quality. Quality comes only from accepted outcome signals and configured
evaluation gates.

## Deployment cooldowns

After the configured retry budget for a deployment is exhausted by a timeout,
rate limit, provider outage, authentication failure, or quota exhaustion, the
orchestrator temporarily isolates that exact provider/account/model deployment.
The default cooldown is 30 seconds and can be configured with
`deployment_cooldown_seconds` in orchestration settings (1–3600 seconds).

Cooldown is scoped to the deployment, not the provider or model family. It
expires automatically, and a successful account health probe clears it early.
The route receipt and Orchestrator preview report `cooling_down` among rejected
candidates. Strict mode may use only an explicit same-model equivalent during a
cooldown; relaxed mode follows the already configured fallback policy. Streams
never switch deployment after their first response byte, even when that byte is
followed by a failure.

## Weighted equivalent deployments

Role bindings may set `equivalent_deployment_weights` to allocate traffic
between the primary deployment and explicitly configured same-model
equivalents. A request ID is deterministically hashed into the configured
positive-weight cohort, so a repeated request ID remains on the same
deployment. The routing receipt and Orchestrator preview expose the resulting
cohort and effective weights.

Weights never broaden model choice: they cannot target a different model,
provider fallback, or arbitrary harness transport. Policy, capability,
account, and cooldown checks run first; an ineligible or cooled deployment
receives zero traffic and appears in the receipt's rejected-candidate list. If
no eligible positive-weight deployment remains, the router retains its existing
reliability-based equivalent selection.

The Orchestrator route preview exposes the request's required capabilities and
provider policy alongside the chosen deployment, weighted cohort (when used),
candidate scores, and rejected-candidate reasons. This lets operators tell
whether a route was constrained by request semantics, policy, temporary
deployment isolation, or the configured allocation.

## Guided Safe Savings

Guided Safe Savings is a read-only operator experience for inspecting
conservative model routing. Enable its discoverability with:

```bash
export CUTCTX_SAFE_SAVINGS_EXPERIENCE=true
cutctx routing status --proxy-url http://127.0.0.1:8787
```

`CUTCTX_SAFE_SAVINGS_EXPERIENCE` controls discoverability only — it surfaces
the `cutctx routing status` command and the dashboard Safe Savings panel. It
never enables routing; routing remains separately opt-in through the existing
preset and orchestrator configuration.

Status is served by the authenticated endpoint
`GET /v1/orchestration/safe-savings/status` (admin key or RBAC `providers.read`
permission). Reading status performs zero provider calls and zero state
writes, and it does not bypass account, capability, or transport proof — it
only reports the routes and the most recent recorded routing decision. The
response includes the active preset, exact route pairs with capability and
transport posture, and the latest applied or retained decision with its
confidence and signals.

Rollback uses the existing authoritative configuration mutation: the dashboard
panel's "Turn Safe Savings off" action (and any API client) posts
`{"orchestrator_mode": "off"}` to the authenticated config endpoint. No new
mutation path is introduced, and new requests then retain their originally
requested models.
